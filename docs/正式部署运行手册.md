# 正式部署运行手册

## 目标

本文记录单机正式部署的推荐流程，用于把当前系统以 SQLite 存储、生产构建、后台 API 服务、Nginx 静态资源和反向代理的方式运行起来。

SQLite 是嵌入式文件数据库，不需要单独启动数据库 daemon。正式环境长期运行的是 Web/API 应用进程；数据库维护推荐通过 `optools.sh data` 统一入口完成，底层仍由 `tools/sqlite-service.sh` 执行。

## 推荐目录

```bash
/opt/my-working-schedule                  # 应用代码
/var/lib/my-working-schedule              # SQLite 数据库和迁移前 JSON 归档
/var/backups/my-working-schedule          # SQLite 备份
/var/log/my-working-schedule-backup.log   # 备份任务日志
```

默认应用目录为 `/opt/my-working-schedule`。在项目根目录执行初始化命令：

```bash
./optools.sh app init
```

该命令会创建 `my-working-schedule` 系统用户和用户组，创建 `/opt/my-working-schedule`、`/var/lib/my-working-schedule`、`/var/backups/my-working-schedule`，修正目录权限，安装 systemd service 文件，并执行 `systemctl daemon-reload` 与 `systemctl enable my-working-schedule`。

`app init` 会自动使用 `command -v npm` 检测当前服务器上的 npm 绝对路径，并写入 systemd 的 `ExecStart`。如果 npm 不在系统默认路径中，可以显式指定：

```bash
OPTOOLS_NPM_BIN=/usr/local/bin/npm ./optools.sh app init
```

如果日志出现 `status=203/EXEC`，通常表示 systemd service 中的 `ExecStart` 指向的可执行文件不存在或不可执行。先执行 `./optools.sh app doctor` 检查 `systemd exec start`，再重新执行 `./optools.sh app init` 让脚本按当前 npm 路径重写 service 文件。

如果源码临时放在 `/root/github/my-working-schedule`，可以先在源码目录执行 `./optools.sh build`，该命令会构建前端并把静态文件安装到 `/opt/my-working-schedule/dist`。

## 依赖安装

OpenCloudOS、CentOS 或 RHEL 系发行版建议先安装构建工具，便于安装 `better-sqlite3`：

```bash
dnf install -y gcc gcc-c++ make python3 nodejs npm
```

如果系统使用 `yum`：

```bash
yum install -y gcc gcc-c++ make python3 nodejs npm
```

进入应用目录后安装依赖：

```bash
cd /opt/my-working-schedule
npm ci --include=dev
```

当前 `start:api` 使用 `tsx` 启动 TypeScript 后端源码，因此正式单机部署仍需要安装 devDependencies。后续如果增加后端编译输出，可以再改为只安装 production dependencies。

## 生产配置

推荐通过 systemd 环境变量配置，至少包含：

```bash
SCHEDULE_STORAGE_DRIVER=sqlite
SCHEDULE_SQLITE_PATH=/var/lib/my-working-schedule/schedule.db
SCHEDULE_BACKUP_PATH=/var/backups/my-working-schedule
SCHEDULE_ADMIN_PASSWORD=请改成真实密码
```

也可以复制生产配置示例：

```bash
cp config/server.production.example.json config/server.local.json
```

然后编辑 `config/server.local.json`，确保：

```json
{
  "storageDriver": "sqlite",
  "sqlitePath": "/var/lib/my-working-schedule/schedule.db",
  "backupPath": "/var/backups/my-working-schedule"
}
```

`storagePath` 可以继续保留，用于记录迁移前 JSON 文件位置；正式运行时以 `sqlitePath` 为主。

## 数据迁移

如果服务器已有 JSON 运行数据，先停止服务并备份源文件：

```bash
cd /opt/my-working-schedule
mkdir -p backups/json-archive
cp data/app-data.local.json backups/json-archive/app-data-before-sqlite-$(date +%Y%m%d-%H%M%S).json
```

执行预检：

```bash
./optools.sh data install
```

迁移到 SQLite：

```bash
./optools.sh data migrate
```

如果目标库已存在且确认要覆盖：

```bash
npm run data:migrate:sqlite -- --overwrite
```

检查数据库：

```bash
./optools.sh data status
./optools.sh data check
```

`check` 应输出 `ok: true`、`integrity: "ok"`，且 `missingTables` 为空数组。

## 构建前端

```bash
npm run build
```

构建并安装静态文件：

```bash
./optools.sh build
```

该命令会执行 `npm run build`，然后把 `dist` 安装到 `/opt/my-working-schedule/dist`。构建完成后，Nginx 使用 `/opt/my-working-schedule/dist` 作为静态资源目录。

## API 服务

项目提供生产 API 启动脚本：

```bash
npm run start:api
```

该命令只启动 Express API，不启动 Vite 开发服务，也不使用 `server/dev-api-watch.mjs`。

systemd 示例位于：

```bash
deploy/systemd/my-working-schedule.service.example
```

部署时初始化 systemd 服务并检查用户、路径和密码：

```bash
./optools.sh app init
./optools.sh app doctor
./optools.sh app start
./optools.sh app status
```

查看日志：

```bash
./optools.sh app logs -f
```

## Nginx 反向代理

Nginx 示例位于：

```bash
deploy/nginx/my-working-schedule.conf.example
```

推荐使用项目脚本安装和配置 Nginx：

```bash
./optools.sh nginx install
```

脚本会在缺少 nginx 时自动调用 `dnf install -y nginx`、`yum install -y nginx` 或 `apt-get install -y nginx`；随后创建 `/etc/nginx/conf.d`，复制 `deploy/nginx/my-working-schedule.conf.example` 到 `/etc/nginx/conf.d/my-working-schedule.conf`，执行 `nginx -t`，并通过 systemd 启用和 reload nginx。

如果只想复制配置并验证，但暂不 reload 服务：

```bash
./optools.sh nginx configure --no-reload
```

常用检查命令：

```bash
./optools.sh nginx status
./optools.sh nginx test
./optools.sh nginx reload
```

该配置将前端静态文件指向 `/opt/my-working-schedule/dist`，并把 `/api/` 反向代理到 `http://127.0.0.1:3001/api/`。

## 健康检查

服务启动后检查 API：

```bash
./optools.sh app health
```

通过 Nginx 检查：

```bash
curl -I http://127.0.0.1/
curl -fsS http://127.0.0.1/api/health
```

页面验证：

1. 打开站点首页。
2. 输入管理密码进入编辑模式。
3. 查看排班、周统计、月结与奖金页面。
4. 保存一条测试配置或排班，再确认 `stat /var/lib/my-working-schedule/schedule.db` 修改时间更新。
5. 执行 `./optools.sh data check`。

## 备份

手动备份：

```bash
./optools.sh data backup
```

建议在月结、系统升级和服务器迁移前手动备份一次。

定时任务示例位于：

```bash
deploy/cron/my-working-schedule-backup.cron.example
```

安装示例：

```bash
crontab deploy/cron/my-working-schedule-backup.cron.example
```

备份后可查看：

```bash
ls -lh /var/backups/my-working-schedule
```

## 恢复

恢复前先停止 API 服务：

```bash
./optools.sh app stop
CONFIRM_RESTORE=yes ./optools.sh data restore <backup-file>
./optools.sh data check
./optools.sh app start
./optools.sh app status
```

`<backup-file>` 可以是 `/var/backups/my-working-schedule` 下的文件名，也可以是绝对路径。

## 上线检查清单

- Node.js、npm、gcc、make、python3 已安装。
- `./optools.sh app init` 执行成功，系统用户、目录权限和 systemd service 已初始化。
- `./optools.sh app doctor` 通过。
- `./optools.sh nginx install` 执行成功，`nginx -t` 通过。
- `npm ci --include=dev` 完成，无缺失 `better-sqlite3`、`tsx`、`vite` 等依赖。
- `config/server.local.json` 或 systemd 环境变量已配置 SQLite 和管理密码。
- `./optools.sh data install` 输出的 `sqlitePath`、`backupPath` 正确。
- `./optools.sh data check` 通过。
- `./optools.sh build` 成功生成并安装 `/opt/my-working-schedule/dist`。
- `npm run start:api` 可启动 API。
- `./optools.sh app status` 正常。
- `/api/health` 可访问。
- 首页、排班、系统配置、周统计、月结与奖金页面可正常读取。
- `./optools.sh data backup` 可生成备份文件。
- `./optools.sh doctor` 通过。

## 回滚策略

如果 SQLite 切换后发现数据异常：

1. 停止 `my-working-schedule` 服务。
2. 保留当前 SQLite 文件和备份目录。
3. 用迁移前 JSON 归档重新执行迁移，或恢复切换前 SQLite 备份。
4. 执行 `./optools.sh data check`。
5. 重新启动服务并核对关键页面。

旧 JSON 文件在 SQLite 稳定运行前不要删除。正式生产数据以 SQLite 为准，JSON 只作为迁移源、归档和人工核对材料。
