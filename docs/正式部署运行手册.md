# 正式部署运行手册

## 目标

本文记录单机正式部署的推荐流程，用于把当前系统以 SQLite 存储、生产构建、后台 API 服务、Nginx 静态资源和反向代理的方式运行起来。

SQLite 是嵌入式文件数据库，不需要单独启动数据库 daemon。正式环境长期运行的是 Web/API 应用进程；数据库维护通过 `tools/sqlite-service.sh` 完成。

## 推荐目录

```bash
/opt/my-working-schedule                  # 应用代码
/var/lib/my-working-schedule              # SQLite 数据库和迁移前 JSON 归档
/var/backups/my-working-schedule          # SQLite 备份
/var/log/my-working-schedule-backup.log   # 备份任务日志
```

建议创建专用用户运行服务：

```bash
useradd --system --create-home --home-dir /var/lib/my-working-schedule --shell /sbin/nologin my-working-schedule
mkdir -p /opt/my-working-schedule /var/lib/my-working-schedule /var/backups/my-working-schedule
chown -R my-working-schedule:my-working-schedule /opt/my-working-schedule /var/lib/my-working-schedule /var/backups/my-working-schedule
```

## 依赖安装

OpenCloudOS、CentOS 或 RHEL 系发行版建议先安装构建工具，便于安装 `better-sqlite3`：

```bash
dnf install -y gcc gcc-c++ make python3 nodejs npm nginx
```

如果系统使用 `yum`：

```bash
yum install -y gcc gcc-c++ make python3 nodejs npm nginx
```

进入应用目录后安装依赖：

```bash
cd /opt/my-working-schedule
npm ci --include=dev
```

当前 `start:api` 使用 `tsx` 启动 TypeScript 后端源码，因此正式单机部署仍需要安装 devDependencies。后续如果增加后端编译输出，可以再改为只安装 production dependencies。

## 生产配置

推荐通过 systemd 环境变量配置，至少包含：

```bash
SCHEDULE_STORAGE_DRIVER=sqlite
SCHEDULE_SQLITE_PATH=/var/lib/my-working-schedule/schedule.db
SCHEDULE_BACKUP_PATH=/var/backups/my-working-schedule
SCHEDULE_ADMIN_PASSWORD=请改成真实密码
```

也可以复制生产配置示例：

```bash
cp config/server.production.example.json config/server.local.json
```

然后编辑 `config/server.local.json`，确保：

```json
{
  "storageDriver": "sqlite",
  "sqlitePath": "/var/lib/my-working-schedule/schedule.db",
  "backupPath": "/var/backups/my-working-schedule"
}
```

`storagePath` 可以继续保留，用于记录迁移前 JSON 文件位置；正式运行时以 `sqlitePath` 为主。

## 数据迁移

如果服务器已有 JSON 运行数据，先停止服务并备份源文件：

```bash
cd /opt/my-working-schedule
mkdir -p backups/json-archive
cp data/app-data.local.json backups/json-archive/app-data-before-sqlite-$(date +%Y%m%d-%H%M%S).json
```

执行预检：

```bash
npm run data:preflight
./tools/sqlite-service.sh install
```

迁移到 SQLite：

```bash
./tools/sqlite-service.sh migrate
```

如果目标库已存在且确认要覆盖：

```bash
npm run data:migrate:sqlite -- --overwrite
```

检查数据库：

```bash
./tools/sqlite-service.sh status
./tools/sqlite-service.sh check
```

`check` 应输出 `ok: true`、`integrity: "ok"`，且 `missingTables` 为空数组。

## 构建前端

```bash
npm run build
```

构建完成后，Nginx 使用 `/opt/my-working-schedule/dist` 作为静态资源目录。

## API 服务

项目提供生产 API 启动脚本：

```bash
npm run start:api
```

该命令只启动 Express API，不启动 Vite 开发服务，也不使用 `server/dev-api-watch.mjs`。

systemd 示例位于：

```bash
deploy/systemd/my-working-schedule.service.example
```

部署时复制并检查用户、路径和密码：

```bash
cp deploy/systemd/my-working-schedule.service.example /etc/systemd/system/my-working-schedule.service
systemctl daemon-reload
systemctl enable my-working-schedule
systemctl start my-working-schedule
systemctl status my-working-schedule
```

查看日志：

```bash
journalctl -u my-working-schedule -f
```

## Nginx 反向代理

Nginx 示例位于：

```bash
deploy/nginx/my-working-schedule.conf.example
```

部署时复制到 Nginx 配置目录：

```bash
cp deploy/nginx/my-working-schedule.conf.example /etc/nginx/conf.d/my-working-schedule.conf
nginx -t
systemctl reload nginx
```

该配置将前端静态文件指向 `/opt/my-working-schedule/dist`，并把 `/api/` 反向代理到 `http://127.0.0.1:3001/api/`。

## 健康检查

服务启动后检查 API：

```bash
curl -fsS http://127.0.0.1:3001/api/health
```

通过 Nginx 检查：

```bash
curl -I http://127.0.0.1/
curl -fsS http://127.0.0.1/api/health
```

页面验证：

1. 打开站点首页。
2. 输入管理密码进入编辑模式。
3. 查看排班、周统计、月结与奖金页面。
4. 保存一条测试配置或排班，再确认 `stat /var/lib/my-working-schedule/schedule.db` 修改时间更新。
5. 执行 `./tools/sqlite-service.sh check`。

## 备份

手动备份：

```bash
./tools/sqlite-service.sh backup
```

建议在月结、系统升级和服务器迁移前手动备份一次。

定时任务示例位于：

```bash
deploy/cron/my-working-schedule-backup.cron.example
```

安装示例：

```bash
crontab deploy/cron/my-working-schedule-backup.cron.example
```

备份后可查看：

```bash
ls -lh /var/backups/my-working-schedule
```

## 恢复

恢复前先停止 API 服务：

```bash
systemctl stop my-working-schedule
CONFIRM_RESTORE=yes ./tools/sqlite-service.sh restore <backup-file>
./tools/sqlite-service.sh check
systemctl start my-working-schedule
systemctl status my-working-schedule
```

`<backup-file>` 可以是 `/var/backups/my-working-schedule` 下的文件名，也可以是绝对路径。

## 上线检查清单

- Node.js、npm、gcc、make、python3、nginx 已安装。
- `npm ci --include=dev` 完成，无缺失 `better-sqlite3`、`tsx`、`vite` 等依赖。
- `config/server.local.json` 或 systemd 环境变量已配置 SQLite 和管理密码。
- `npm run data:preflight` 输出的 `sqlitePath`、`backupPath` 正确。
- `./tools/sqlite-service.sh check` 通过。
- `npm run build` 成功生成 `dist`。
- `npm run start:api` 可启动 API。
- `systemctl status my-working-schedule` 正常。
- `nginx -t` 通过。
- `/api/health` 可访问。
- 首页、排班、系统配置、周统计、月结与奖金页面可正常读取。
- `./tools/sqlite-service.sh backup` 可生成备份文件。

## 回滚策略

如果 SQLite 切换后发现数据异常：

1. 停止 `my-working-schedule` 服务。
2. 保留当前 SQLite 文件和备份目录。
3. 用迁移前 JSON 归档重新执行迁移，或恢复切换前 SQLite 备份。
4. 执行 `./tools/sqlite-service.sh check`。
5. 重新启动服务并核对关键页面。

旧 JSON 文件在 SQLite 稳定运行前不要删除。正式生产数据以 SQLite 为准，JSON 只作为迁移源、归档和人工核对材料。
