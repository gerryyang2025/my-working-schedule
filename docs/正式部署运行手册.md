# 正式部署运行手册

## 目标

本文记录单机正式部署的推荐流程，用于把当前系统以 SQLite 存储、生产构建、后台 API 服务、Nginx 静态资源和反向代理的方式运行起来。

SQLite 是嵌入式文件数据库，不需要单独启动数据库 daemon。正式环境长期运行的是 Web/API 应用进程；数据库维护推荐通过 `optools.sh data` 统一入口完成，底层仍由 `tools/sqlite-service.sh` 执行。

## 推荐目录

```bash
/opt/my-working-schedule                  # 应用代码
/var/lib/my-working-schedule              # SQLite 数据库和迁移前 JSON 归档
/var/backups/my-working-schedule          # SQLite 备份
/var/log/my-working-schedule-backup.log   # 备份任务日志
```

默认应用目录为 `/opt/my-working-schedule`。推荐始终在源码目录执行一键部署：

```bash
cd /root/github/my-working-schedule
git pull
./optools.sh deploy
```

`deploy` 会创建 `my-working-schedule` 系统用户和用户组，创建 `/opt/my-working-schedule`、`/var/lib/my-working-schedule`、`/var/backups/my-working-schedule`，修正目录权限，安装 systemd service 文件，构建前端，安装 API 运行文件，在 `/opt/my-working-schedule` 安装依赖，检查 SQLite 和 Nginx，最后重启 API 服务并执行健康检查。

`deploy` 会自动寻找 systemd 服务用户可执行的 npm，并写入 systemd 的 `ExecStart`。如果服务器使用特殊 Node.js 安装路径，可以显式指定：

```bash
OPTOOLS_NPM_BIN=/custom/node/bin/npm ./optools.sh deploy
```

如果日志出现 `status=203/EXEC`，通常表示 systemd service 中的 `ExecStart` 指向的可执行文件不存在，或服务用户没有权限执行它。不要把 `/root/.nvm/.../npm` 直接配置给 `my-working-schedule` 这类非 root 服务用户，因为 `/root` 目录通常不可穿透。

推荐把 Node.js 放到服务用户可访问的位置，例如：

```bash
mkdir -p /opt/node-v22.22.0
cp -a /root/.nvm/versions/node/v22.22.0/. /opt/node-v22.22.0/
chown -R root:root /opt/node-v22.22.0
chmod -R a+rX /opt/node-v22.22.0

./optools.sh deploy
```

`deploy` 会自动尝试 `/opt/node-v22.22.0/bin/npm`、`/opt/node/bin/npm`、`/usr/local/bin/npm`、`/usr/bin/npm` 等候选路径。`app doctor` 是 API/systemd 专项检查，会检查 `systemd exec start`，确认当前服务用户实际能运行 `ExecStart`。

## 分步骤排障

如果需要拆开排查，仍可使用下面的单项命令。常规上线优先使用 `./optools.sh deploy`。

## 依赖安装

OpenCloudOS、CentOS 或 RHEL 系发行版建议先安装构建工具，便于安装 `better-sqlite3`：

```bash
dnf install -y gcc gcc-c++ make python3 nodejs npm
```

如果系统使用 `yum`：

```bash
yum install -y gcc gcc-c++ make python3 nodejs npm
```

`deploy` 会自动在 `/opt/my-working-schedule` 安装依赖，不需要手动切换目录。排障时可以手工执行：

```bash
npm --prefix /opt/my-working-schedule ci --include=dev
```

当前 `start:api` 使用 `tsx` 启动 TypeScript 后端源码，因此正式单机部署仍需要安装 devDependencies。后续如果增加后端编译输出，可以再改为只安装 production dependencies。

## 生产配置

推荐通过 systemd 环境变量配置，至少包含：

```bash
SCHEDULE_STORAGE_DRIVER=sqlite
SCHEDULE_SQLITE_PATH=/var/lib/my-working-schedule/schedule.db
SCHEDULE_BACKUP_PATH=/var/backups/my-working-schedule
SCHEDULE_ADMIN_PASSWORD=请改成真实密码
```

登录页默认管理员账号为 `admin`。`SCHEDULE_ADMIN_PASSWORD` 或 `config/server.local.json` 中的 `adminPassword` 用于初始化并刷新该默认管理员密码；SQLite 模式下实际登录密码以哈希形式保存到 `users` 表，不写入排班业务数据。

也可以复制生产配置示例：

```bash
cp config/server.production.example.json config/server.local.json
```

然后编辑 `config/server.local.json`，确保：

```json
{
  "storageDriver": "sqlite",
  "sqlitePath": "/var/lib/my-working-schedule/schedule.db",
  "backupPath": "/var/backups/my-working-schedule"
}
```

`storagePath` 可以继续保留，用于记录迁移前 JSON 文件位置；正式运行时以 `sqlitePath` 为主。

## 数据迁移

如果服务器已有 JSON 运行数据，先停止服务并备份源文件：

```bash
cd /opt/my-working-schedule
mkdir -p backups/json-archive
cp data/app-data.local.json backups/json-archive/app-data-before-sqlite-$(date +%Y%m%d-%H%M%S).json
```

执行预检：

```bash
./optools.sh data install
```

迁移到 SQLite：

```bash
./optools.sh data migrate
```

如果目标库已存在且确认要覆盖：

```bash
npm run data:migrate:sqlite -- --overwrite
```

检查数据库：

```bash
./optools.sh data status
./optools.sh data check
```

`check` 应输出 `ok: true`、`integrity: "ok"`，且 `missingTables` 为空数组。

## 构建并安装应用

在源码目录执行构建并安装生产运行目录：

```bash
./optools.sh build
```

该命令会执行 `npm run build`，然后把 `dist` 安装到 `/opt/my-working-schedule/dist`，同时把 API 运行需要的 `package.json`、`package-lock.json`、`server/`、`src/`、`tsconfig*.json` 和 `config/server.production.example.json` 安装到 `/opt/my-working-schedule`。构建完成后，Nginx 使用 `/opt/my-working-schedule/dist` 作为静态资源目录，systemd 使用 `/opt/my-working-schedule` 作为 API 工作目录。

如果 systemd 日志出现以下错误，说明这一步还没有执行或安装目录被清空：

```text
npm error enoent Could not read package.json
npm error path /opt/my-working-schedule/package.json
```

修复方法是在源码目录重新执行：

```bash
./optools.sh deploy
./optools.sh app status
```

如果 systemd 日志出现以下错误，说明 API 端口被其他进程占用：

```text
Error: listen EADDRINUSE: address already in use 127.0.0.1:3001
```

先确认占用者：

```bash
ss -ltnp | grep ':3001'
lsof -nP -iTCP:3001 -sTCP:LISTEN
```

常见原因是仍有开发 daemon 或手工启动的 API 进程。处理示例：

```bash
./optools.sh dev stop
./optools.sh app stop
ss -ltnp | grep ':3001'
./optools.sh deploy
```

## API 服务

项目提供生产 API 启动脚本：

```bash
npm run start:api
```

该命令只启动 Express API，不启动 Vite 开发服务，也不使用 `server/dev-api-watch.mjs`。

systemd 示例位于：

```bash
deploy/systemd/my-working-schedule.service.example
```

部署时初始化 systemd 服务并检查用户、路径和密码：

```bash
./optools.sh app init
./optools.sh app doctor
./optools.sh app start
./optools.sh app status
```

查看日志：

```bash
./optools.sh app logs -f
```

## Nginx 反向代理

Nginx 示例位于：

```bash
deploy/nginx/my-working-schedule.conf.example
```

推荐使用项目脚本安装和配置 Nginx：

```bash
./optools.sh nginx install
```

脚本会在缺少 nginx 时自动调用 `dnf install -y nginx`、`yum install -y nginx` 或 `apt-get install -y nginx`；随后创建 `/etc/nginx/conf.d`，复制 `deploy/nginx/my-working-schedule.conf.example` 到 `/etc/nginx/conf.d/my-working-schedule.conf`，执行 `nginx -t`，并通过 systemd 启用和 reload nginx。

如果只想复制配置并验证，但暂不 reload 服务：

```bash
./optools.sh nginx configure --no-reload
```

常用检查命令：

```bash
./optools.sh nginx status
./optools.sh nginx test
./optools.sh nginx reload
```

该配置将前端静态文件指向 `/opt/my-working-schedule/dist`，并把 `/api/` 反向代理到 `http://127.0.0.1:3001/api/`。

## 健康检查

服务启动后检查 API：

```bash
./optools.sh app health
```

通过 Nginx 检查：

```bash
curl -I http://127.0.0.1/
curl -fsS http://127.0.0.1/api/health
```

页面验证：

1. 打开站点首页。
2. 使用 `admin` 和已配置的管理员密码登录。
3. 查看排班、周统计、月结与奖金页面。
4. 打开系统配置，确认账号和审计日志页面可见。
5. 新增一个 `viewer` 测试账号，退出后确认该账号可登录但不能编辑排班。
6. 使用测试账号修改自己的密码，确认修改后需要重新登录。
7. 保存一条测试配置或排班，再确认 `stat /var/lib/my-working-schedule/schedule.db` 修改时间更新。
8. 执行 `./optools.sh data check`。

## 账号维护

除默认 `admin` 外，其他账号通过 Web 管理页面维护：

1. 使用 `admin` 和当前管理员密码登录。
2. 点击右上角“配置”。
3. 进入“账号”tab。
4. 点击“新增账号”。
5. 填写账号、显示名、角色、启用状态和初始密码。
6. 点击“保存账号”。
7. 让新用户使用初始密码登录，并建议首次登录后点击“修改密码”改成个人密码。

当前支持三类角色：

- `系统管理员`：可维护人员、班次、节假日、账号、排班和月结，并可查看审计日志。
- `排班管理员`：可维护排班和月结，不能维护人员、班次、节假日、账号和审计日志。
- `只读查看`：只能查看排班、周统计、月结与奖金等信息，不能保存修改。

如果新增账号无法登录，先确认账号处于“启用”状态，再确认密码是否已正确设置或重置。

## 备份

手动备份：

```bash
./optools.sh data backup
```

建议在月结、系统升级和服务器迁移前手动备份一次。

定时任务示例位于：

```bash
deploy/cron/my-working-schedule-backup.cron.example
```

安装示例：

```bash
crontab deploy/cron/my-working-schedule-backup.cron.example
```

备份后可查看：

```bash
ls -lh /var/backups/my-working-schedule
```

## 恢复

恢复前先停止 API 服务：

```bash
./optools.sh app stop
CONFIRM_RESTORE=yes ./optools.sh data restore <backup-file>
./optools.sh data check
./optools.sh app start
./optools.sh app status
```

`<backup-file>` 可以是 `/var/backups/my-working-schedule` 下的文件名，也可以是绝对路径。

## 上线检查清单

- Node.js、npm、gcc、make、python3 已安装。
- `./optools.sh deploy` 执行成功。
- `./optools.sh app init` 执行成功，系统用户、目录权限和 systemd service 已初始化。
- `./optools.sh app doctor` API/systemd 专项检查通过。
- `./optools.sh nginx install` 执行成功，`nginx -t` 通过。
- `npm --prefix /opt/my-working-schedule ci --include=dev` 完成，无缺失 `better-sqlite3`、`tsx`、`vite` 等依赖。
- `config/server.local.json` 或 systemd 环境变量已配置 SQLite 和默认管理员登录密码。
- `./optools.sh data install` 输出的 `sqlitePath`、`backupPath` 正确。
- `./optools.sh data check` 通过。
- `./optools.sh build` 成功生成并安装 `/opt/my-working-schedule/dist`、`/opt/my-working-schedule/package.json`、`/opt/my-working-schedule/server/index.ts`。
- `npm run start:api` 可启动 API。
- `./optools.sh app status` 正常。
- `/api/health` 可访问。
- 首页、排班、系统配置、周统计、月结与奖金页面可正常读取。
- `./optools.sh data backup` 可生成备份文件。
- `./optools.sh doctor` 完整生产运行体检通过。

## 回滚策略

如果 SQLite 切换后发现数据异常：

1. 停止 `my-working-schedule` 服务。
2. 保留当前 SQLite 文件和备份目录。
3. 用迁移前 JSON 归档重新执行迁移，或恢复切换前 SQLite 备份。
4. 执行 `./optools.sh data check`。
5. 重新启动服务并核对关键页面。

旧 JSON 文件在 SQLite 稳定运行前不要删除。正式生产数据以 SQLite 为准，JSON 只作为迁移源、归档和人工核对材料。
