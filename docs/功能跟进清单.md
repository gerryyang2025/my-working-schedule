# 功能跟进清单

## 文档状态

本文用于统一记录当前系统已完成功能、部分完成能力和后续待实现事项，便于后续排期、验收和继续优化。

更新时间：2026-06-18。

状态说明：

- **已完成**：当前系统已有可用实现，并有基础测试或实际验证支撑。
- **部分完成**：已有核心能力，但与产品需求或正式环境要求相比仍有缺口。
- **待实现**：尚未实现，或仅停留在方案文档中。

## 1. 已完成功能

### 1.1 排班工作台

- 支持按选中日期定位所在自然周，并只展示当前周 7 天。
- 支持上一周、下一周、本周切换。
- 支持班次画笔，点击排班格快速录入班次。
- 支持点击排班格打开弹窗，单人单日最多两个班次。
- 支持人员列固定，移动端横向滚动时仍能看到人员信息。
- 人员列显示姓名和工号。
- 支持停用人员保留历史排班显示，避免历史数据丢失。
- 支持周末、节假日和节日名称视觉标记。
- 支持移动端浏览和移动端快速点格子排班。

### 1.2 人员、班次、节假日配置

- 支持人员维护：工号、姓名、类型、管理员标记、启用状态、排序。
- 支持人员类型：护士、文员、护士长。
- 支持文员显示在排班表中，并按护士相同规则参与统计。
- 支持班次维护：班次名称、简称、颜色、是否计出勤、系数、启用状态、排序。
- 支持节假日维护：日期、名称、是否影响满勤。
- 支持管理密码进入编辑模式后再保存配置。

### 1.3 周统计

- 支持自然周统计。
- 支持按班次而不是自然日统计出勤。
- 支持节假日扣减满勤标准。
- 支持统计出勤班次、满勤标准、加班班次和总系数。
- 支持护士长出勤和加班按相同排班规则统计。
- 支持护士长绩效系数显示为“护士长绩效单独核算”。
- 移动端周统计按人员一行展示，便于查看。

### 1.4 月度汇总、奖金分配和月结

- 支持月度汇总：月出勤班次、累计加班班次、月总系数。
- 支持月结与奖金独立 tab 展示，避免首页信息过多。
- 支持输入奖金总额并按普通人员月总系数自动分配奖金。
- 支持护士长排除普通奖金池，并显示单独核算说明。
- 支持确认月结后保存月结快照。
- 支持已月结月份锁定排班，禁止继续修改该月排班记录。
- 支持取消月结后恢复该月可编辑。
- 支持按月范围进行临时汇总和奖金试算，不保存月结，不锁定排班。
- 支持月结与奖金中显示累计加班班次。

### 1.5 打印、PDF 和移动端输出

- 支持打印周表。
- 支持打印月表。
- 支持月表中输出月度汇总。
- 支持已月结月份在月表中输出奖金分配快照。
- 打印中的人员信息显示姓名和工号。
- 打印中的排班信息使用彩色文字，不再使用外框样式。
- 支持生成 PDF。
- 手机端支持生成或分享 PDF。
- 手机端隐藏重复的系统打印按钮，仅保留更可用的生成或分享 PDF 入口。

### 1.6 开发运行和基础配置

- 支持 `npm run dev` 启动开发服务。
- 支持 `optools.sh` 以 daemon 方式启动、停止、重启和查看开发服务状态。
- 支持 API 和 Web 监听真实 IP，可供局域网访问。
- 支持服务端配置文件 `config/server.local.json`。
- 管理员密码从环境变量或服务端配置文件读取，不写入业务数据文件。
- 当前开发期业务数据写入 `data/app-data.local.json`，仓库种子数据保留在 `data/app-data.json`。

## 2. 部分完成能力

### 2.1 多维时间视图

产品需求提出按年、月、日、周切换视图。当前已支持日期选择、周切换、月表打印和月结按月统计，但主工作台仍以周视图为核心，尚未提供完整的月视图或年视图浏览。

### 2.2 绩效系数配置

当前支持按班次配置系数，并在统计中按班次系数累加。产品需求中提到的岗位系数、辅班等更细分类目尚未独立建模；护士长绩效目前只做“单独核算”标记，未实现护士长绩效算法。

### 2.3 班次分类

当前班次支持启用、排序、颜色和系数，但没有“常用班次 / 不常用班次 / 请假类 / 培训类”等分类。产品需求中提到的进修、病假、产假等不常用班次可以手动新增为普通班次，但缺少分类展示和管理。

### 2.4 出勤盈亏展示

当前周统计和月度汇总已展示出勤班次、满勤标准、加班班次和总系数，但没有单独展示“出勤盈亏”列，例如 `-1 / 0 / +1`。

### 2.5 自动汇总

当前统计和奖金分配是实时计算或由用户手动确认月结。产品需求中“每周五自动汇总”“每月最后一天自动汇总”尚未实现为自动任务或自动生成草稿。

### 2.6 正式存储

当前仍使用 JSON 文件作为开发期存储。正式环境存储已形成 SQLite 优先方案，见 [正式环境存储优化方案](./正式环境存储优化方案.md)，但尚未实现 SQLite 存储、迁移脚本、备份恢复和导入导出。

## 3. 待实现功能

### 3.1 P0：正式化基础能力

- SQLite 存储实现。
- JSON 到 SQLite 的迁移脚本，确保当前 `data/app-data.local.json` 中真实用户数据不丢失。
- SQLite 到 JSON 的导出能力。
- 自动备份和手动备份命令。
- 数据恢复流程和恢复验证。
- 正式账号体系，替代单一管理密码。
- 密码哈希保存、登录会话、退出登录和会话过期。
- 角色权限控制，例如护士长、普通查看人员、系统管理员。
- 操作审计日志，记录排班、配置、月结和取消月结等关键操作。
- 正式部署方案，包括生产构建、进程管理、反向代理、HTTPS、日志轮转和防火墙说明。

### 3.2 P1：业务规则补齐

- 出勤盈亏统计列。
- 班次分类：常用、不常用、请假、培训等。
- 护士长绩效单独核算规则。
- 岗位系数和班次系数的组合规则评估。
- 周汇总和月汇总自动生成草稿或自动提醒。
- 奖金分配结果导出。
- 月结前数据检查，例如未排班人员、异常双班、禁用班次引用。

### 3.3 P2：排班效率与体验优化

- 复制上一周排班。
- 按人员复制排班。
- 批量清空某周或某人排班。
- 批量设置休息或办公班。
- 排班冲突或异常提醒。
- 更多打印模板，例如奖金分配单独打印模板。
- 更完整的移动端排班操作反馈。
- 数据导入导出页面。

## 5. 建议下一步

建议下一阶段优先处理 P0 正式化基础能力：

1. 先设计 SQLite 存储和 JSON 迁移脚本。
2. 再实现备份、恢复和导入导出。
3. 然后实现账号权限和审计日志。
4. 最后整理正式部署脚本和运维文档。

完成 P0 后，系统才更适合作为正式环境长期使用。
