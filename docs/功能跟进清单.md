# 功能跟进清单

## 文档状态

本文用于统一记录当前系统已完成功能、部分完成能力和后续待实现事项，便于后续排期、验收和继续优化。

更新时间：2026-06-21。

状态说明：

- **已完成**：当前系统已有可用实现，并有基础测试或实际验证支撑。
- **部分完成**：已有核心能力，但与产品需求或正式环境要求相比仍有缺口。
- **待实现**：尚未实现，或仅停留在方案文档中。

## 1. 已完成功能

### 1.1 排班工作台

- 支持按选中日期定位所在自然周，并只展示当前周 7 天。
- 支持上一周、下一周、本周切换。
- 支持班次画笔，点击排班格快速录入班次。
- 支持点击排班格打开弹窗，单人单日最多两个班次。
- 支持复制上一周排班；当前周为空时直接复制，当前周已有排班时可选择跳过已有格子或覆盖已有格子。
- 支持人员列固定，移动端横向滚动时仍能看到人员信息。
- 人员列显示姓名和工号。
- 支持停用人员保留历史排班显示，避免历史数据丢失。
- 支持周末、节假日和节日名称视觉标记。
- 支持移动端浏览和移动端快速点格子排班。

### 1.2 人员、班次、节假日配置

- 支持人员维护：工号、姓名、类型、管理员标记、启用状态、排序。
- 支持人员类型：护士、文员、护士长。
- 支持文员显示在排班表中，并按护士相同规则参与统计。
- 支持班次维护：班次名称、简称、颜色、是否计出勤、系数、启用状态、排序。
- 支持节假日维护：日期、名称、是否影响满勤。
- 支持登录后按角色保存配置。

### 1.3 周统计

- 支持自然周统计。
- 支持按班次而不是自然日统计出勤。
- 支持节假日扣减满勤标准。
- 支持统计出勤班次、满勤标准、加班班次和总系数。
- 支持出勤盈亏统计，按 `出勤班次 - 满勤标准` 显示 `+1 / 0 / -1`。
- 支持护士长出勤和加班按相同排班规则统计。
- 支持护士长绩效系数显示为“护士长绩效单独核算”。
- 移动端周统计按人员一行展示，便于查看。

### 1.4 月度汇总、奖金分配和月结

- 支持月度汇总：月出勤班次、满勤标准、出勤盈亏、累计加班班次、月总系数。
- 支持月结与奖金独立 tab 展示，避免首页信息过多。
- 支持输入奖金总额并按普通人员月总系数自动分配奖金。
- 支持护士长排除普通奖金池，并显示单独核算说明。
- 支持确认月结后保存月结快照。
- 支持已月结月份锁定排班，禁止继续修改该月排班记录。
- 支持取消月结后恢复该月可编辑。
- 支持按月范围进行临时汇总和奖金试算，不保存月结，不锁定排班。
- 支持月结与奖金中显示满勤标准、出勤盈亏和累计加班班次。
- 支持确认月结前进行数据检查，提示未排班、出勤不足、异常双班、缺失班次、停用班次引用和停用人员排班；检查结果仅提醒，不阻止用户继续月结。
- 支持奖金分配结果导出 CSV；单月导出当前月结果，已月结月份优先导出月结快照，临时时间范围导出当前试算结果。

### 1.5 打印、PDF 和移动端输出

- 支持打印周表。
- 支持打印月表。
- 支持月表中输出月度汇总。
- 支持已月结月份在月表中输出奖金分配快照。
- 支持周表、月表和奖金分配快照中输出满勤标准和出勤盈亏。
- 打印中的人员信息显示姓名和工号。
- 打印中的排班信息使用彩色文字，不再使用外框样式。
- 支持生成 PDF。
- 手机端支持生成或分享 PDF。
- 手机端隐藏重复的系统打印按钮，仅保留更可用的生成或分享 PDF 入口。
- 支持 PDF 生成时完整展开打印内容，并按横向和纵向自动分页，避免周表、月表内容被预览滚动区域或单页宽度截断。

### 1.6 开发运行和基础配置

- 支持 `npm run dev` 启动开发服务。
- 支持 `optools.sh` 以 daemon 方式启动、停止、重启和查看开发服务状态。
- 支持 API 和 Web 监听真实 IP，可供局域网访问。
- 支持 `optools.sh config`、`config paths`、`config server` 查看主要运行配置、关键路径和脱敏后的服务端配置。
- 支持服务端配置文件 `config/server.local.json`。
- 管理员密码从环境变量或服务端配置文件读取，不写入业务数据文件。
- 当前开发期和正式部署均使用 SQLite 作为运行时存储，不再保留 JSON 运行时存储、旧 JSON 导入或 SQLite 导出 JSON 工具。

### 1.7 正式存储基础能力

- SQLite 存储与备份：已实现基础存储、初始化、检查、备份、恢复和 Linux 维护脚本；正式部署通过 `./optools.sh deploy` 统一完成构建、依赖安装、SQLite/Nginx/logrotate 检查、服务重启和 API 健康检查等待。
- SQLite 月结快照已持久化满勤标准和出勤盈亏，数据检查可识别旧库缺少相关列的情况，并通过初始化迁移补齐。
- 正式部署运行手册：已补充生产配置示例、API 生产启动脚本、systemd 示例、Nginx 安装配置脚本、Nginx 示例、SQLite 定时备份示例和上线检查清单；当前阶段推荐 HTTP + IP 访问，后续申请正式域名后再启用 HTTPS。
- 支持 `optools.sh nginx configure-https` 根据已有证书生成 HTTPS Nginx 配置，并执行 `nginx -t` 验证。
- 支持 `optools.sh logrotate install/status/test` 安装和 dry-run 验证日志轮转配置。
- 支持 `optools.sh firewall status/guide` 检查常见 Linux 防火墙工具，并输出 80/443 和 API 3001 暴露策略建议。
- `optools.sh doctor` 已补充日志轮转、防火墙和可选 HTTPS 配置检查；`deploy` 会先安装并 dry-run 验证 logrotate，避免部署后立即执行 `doctor` 时因日志轮转未初始化而失败。

### 1.8 账号权限与审计基础

- 支持登录页面，未登录时先进入登录页。
- 支持默认管理员账号 `admin`，密码来自 `SCHEDULE_ADMIN_PASSWORD` 或服务端配置文件。
- 支持管理员密码哈希保存到 SQLite，重复初始化会按最新配置刷新默认管理员密码。
- 支持登录会话、退出登录和会话过期。
- 支持基础角色权限：`admin` 可维护配置和排班，`scheduler` 可维护排班和月结，`viewer` 只读。
- 支持关键操作审计：登录、退出、配置变更、排班变更、月结和取消月结。
- 支持账号维护页面：新增账号、修改显示名、调整角色、启停账号、重置密码。
- 支持登录用户修改自己的密码，修改成功后需要重新登录。
- 支持审计日志查看页面，并可按账号、操作类型和关键词筛选。
- 已在正式部署运行手册中补充非 `admin` 账号的添加方式和三类角色说明。
- 登录后用户信息显示已避免 `系统管理员 · 系统管理员` 一类重复文案。
- 保留 `/api/admin/session` 兼容入口，便于旧客户端或测试过渡。
- 支持账号与人员档案一对一可选绑定，绑定入口位于“系统配置 > 账号”。
- 支持排班管理员账号配置可管理人员名单；排班和月结写入按名单做后端权限校验。
- 支持所有登录账号查看全科排班，`viewer` 保持只读。

## 2. 部分完成能力

### 2.1 多维时间视图

产品需求提出按年、月、日、周切换视图。当前已支持日期选择、周切换、月表打印和月结按月统计，但主工作台仍以周视图为核心，尚未提供完整的月视图或年视图浏览。

### 2.2 绩效系数配置

当前支持按班次配置系数，并在统计中按班次系数累加。产品需求中提到的岗位系数、辅班等更细分类目尚未独立建模；护士长绩效目前只做“单独核算”标记，未实现护士长绩效算法。

### 2.3 班次分类

暂不实现独立分类模型。当前班次配置已支持自定义新增班次，并可维护简称、颜色、是否计出勤、系数、启用状态和排序；进修、病假、产假、培训、办公等需求可通过自定义班次覆盖。后续只有当班次数量明显增加、画笔分组管理成为真实痛点时，再评估“常用 / 请假 / 培训 / 不常用”等分类展示。

### 2.4 自动汇总

当前统计和奖金分配是实时计算或由用户手动确认月结。产品需求中“每周五自动汇总”“每月最后一天自动汇总”尚未实现为自动任务或自动生成草稿。

### 2.5 细粒度账号权限

当前已实现账号绑定人员、排班管理员可管理人员名单，以及排班和月结写入范围校验。所有登录账号仍可查看全科排班；个人只看自己排班、科室或小组模型暂不实现，作为后续组织权限模型评估。

## 3. 待实现功能

### 3.1 P0：正式化基础能力

- 目标服务器二次联调，当前阶段覆盖 HTTP + IP、日志轮转、防火墙、安全组和备份恢复流程；HTTPS 证书切换留到后续正式域名阶段。
- 后续更复杂组织权限评估，例如个人只看自己排班、科室、小组或护士长辖区模型。

### 3.2 P1：业务规则补齐

- 护士长绩效单独核算规则。
- 岗位系数和班次系数的组合规则评估。
- 周汇总和月汇总自动生成草稿或自动提醒。

### 3.3 P2：排班效率与体验优化

- 按人员复制排班。
- 批量清空某周或某人排班。
- 批量设置休息或办公班。
- 排班冲突或异常提醒。
- 更多打印模板，例如奖金分配单独打印模板。
- 更完整的移动端排班操作反馈。
- 数据导入导出页面。

## 5. 建议下一步

建议下一阶段优先处理剩余 P0 正式化基础能力：

1. 在目标服务器按 [正式部署运行手册](./正式部署运行手册.md) 完成二次联调，确认 HTTP + IP、systemd、Nginx、日志轮转、防火墙、安全组、备份和恢复流程。
2. 再评估个人可见性、护士长辖区权限等细粒度权限。
3. 最后进入 P1 业务规则补齐。

完成 P0 后，系统才更适合作为正式环境长期使用。
