# 正式环境存储方案

## 文档状态

本文记录当前正式环境的存储方案和维护边界。系统已经统一为 SQLite 运行时存储，不再保留 JSON 运行时存储、旧 JSON 数据导入或 SQLite 导出 JSON 的工具链。

结论：当前正式环境采用 **SQLite + 自动备份 + 恢复校验**。后续如果出现多科室、多实例或集中部署需求，再评估 PostgreSQL；在此之前不维护多套存储方案。

## 当前存储方案

当前系统使用 Express API 写入 SQLite 文件数据库：

- 默认开发数据库：`data/schedule.db`。
- 推荐正式数据库：`/var/lib/my-working-schedule/schedule.db`。
- 推荐正式备份目录：`/var/backups/my-working-schedule`。
- 管理员初始化密码来自 `SCHEDULE_ADMIN_PASSWORD` 或 `config/server.local.json`，实际账号密码以哈希形式保存到 `users` 表。

SQLite 保存的业务数据包括：

- 人员资料。
- 班次配置。
- 节假日配置。
- 排班记录。
- 月结与奖金快照。
- 账号、会话和审计日志。
- 系统基础设置。

## 维护边界

保留的 SQLite 维护能力：

- 初始化数据库结构：`./optools.sh data init` 或 `npm run data:init:sqlite`。
- 检查数据库完整性：`./optools.sh data check` 或 `npm run data:check:sqlite`。
- 备份数据库：`./optools.sh data backup` 或 `npm run data:backup`。
- 恢复数据库：`CONFIRM_RESTORE=yes ./optools.sh data restore <backup-file>` 或 `npm run data:restore`。
- 生产预检：`./optools.sh data install` 和 `npm run data:preflight`。

已移除的旧方案能力：

- JSON 文件运行时存储。
- 旧 JSON 数据导入 SQLite。
- SQLite 导出 JSON 文件。
- 旧数据源路径环境变量相关入口。

保留 `config/server.local.json` 是因为它是服务配置文件，不是业务数据存储方案。

## 运维建议

正式环境建议使用统一入口：

```bash
./optools.sh deploy
./optools.sh doctor
./optools.sh data status
./optools.sh data check
./optools.sh data backup
CONFIRM_RESTORE=yes ./optools.sh data restore <backup-file>
```

备份建议：

- 月结前手动执行一次 `./optools.sh data backup`。
- 系统升级、服务器迁移、批量修改配置前手动备份。
- 配置定时任务保存备份到 `/var/backups/my-working-schedule`。
- 定期演练恢复流程，恢复后必须执行 `./optools.sh data check`。

恢复流程：

1. 停止 Web/API 服务。
2. 确认备份文件来源和时间点。
3. 执行 `CONFIRM_RESTORE=yes ./optools.sh data restore <backup-file>`。
4. 执行 `./optools.sh data check`。
5. 重启服务并核对首页、排班、系统配置、审计日志和月结页面。

## 后续升级方向

SQLite 适合当前单机部署。以下情况出现时，再评估 PostgreSQL：

- 多台应用服务器同时写入。
- 多科室、多院区集中部署。
- 需要更复杂的统计查询和报表。
- 需要独立数据库权限、审计、灾备和高可用。

升级 PostgreSQL 时，应以当前 SQLite schema 和 API 行为为基准设计迁移，不再恢复 JSON 存储方案。
