# 正式环境存储优化方案

## 文档状态

本文记录正式环境存储演进建议，作为后续从开发期 JSON 文件存储升级到正式持久化能力的依据。

结论：正式环境第一阶段建议采用 **SQLite + 自动备份 + 导入导出**。JSON 文件继续作为开发验证、数据导出和初始化种子格式保留；PostgreSQL 作为未来多科室、多实例或集中部署时的升级方向。

## 1. 当前存储现状

当前系统使用 Express API 写入本地 JSON 文件：

- 仓库种子数据：`data/app-data.json`。
- 开发运行时数据：`data/app-data.local.json`。
- 可通过 `SCHEDULE_DATA_PATH` 或 `config/server.local.json` 中的 `storagePath` 指定其他数据文件。
- 管理员密码不写入业务数据文件，来自 `SCHEDULE_ADMIN_PASSWORD` 或 `config/server.local.json`。

当前 JSON 文件保存的业务数据包括：

- 人员资料。
- 班次配置。
- 节假日配置。
- 排班记录。
- 月结与奖金快照。
- 系统基础设置。

服务端写入时已经采用串行更新和临时文件替换，能降低单进程并发覆盖和半写入损坏风险。但 JSON 仍然不适合作为正式环境主存储。

### `server.local.json` 与 SQLite 迁移关系

`config/server.local.json` 保存的是服务运行配置，不是业务数据。典型字段包括：

- `host`：API 监听地址。
- `port`：API 监听端口。
- `storagePath`：当前业务数据文件路径。
- `adminPassword`：当前单管理员密码。

这些信息后续不建议整体导入 SQLite。原因是：

- `host`、`port` 属于部署环境配置，应继续由配置文件、环境变量或进程管理工具维护。
- `storagePath` 只用于定位当前 JSON 或 SQLite 文件，不是业务表数据。
- `adminPassword` 属于敏感配置，不应以明文形式迁入业务数据库。

后续从 JSON 迁移到 SQLite 时，真正需要导入的是 `storagePath` 指向的业务数据文件，例如 `data/app-data.local.json`。`server.local.json` 只用于告诉迁移脚本去哪里读取源数据、后续 SQLite 数据库写到哪里。

如果后续实现账号体系，可以提供一次性初始化逻辑：

- 从 `SCHEDULE_ADMIN_PASSWORD` 或 `server.local.json` 读取当前管理员密码。
- 创建默认管理员账号。
- 将密码哈希写入 SQLite 的 `users` 表。
- 初始化完成后，不再依赖明文 `adminPassword` 作为长期登录凭据。

这样既能平滑过渡，又避免把明文密码长期保存到 SQLite 中。

## 2. JSON 在正式环境的风险

JSON 适合开发验证，但正式环境存在以下风险：

- 本地运行文件被 `.gitignore` 忽略，服务器迁移、重新 clone、容器重建时容易遗漏。
- 没有数据库事务和约束，复杂写入依赖应用层保证。
- 没有内置历史版本、审计日志和回滚能力。
- 文件误删或路径变更后，页面会表现为数据丢失或回到种子数据。
- 后续账号权限、月结锁定、奖金分配、审计追踪和统计查询会越来越难维护。
- 多进程或多实例部署时，当前进程内写入队列无法防止跨进程覆盖。

因此，JSON 不建议继续作为正式环境主存储，只建议保留为：

- 开发环境默认存储。
- 种子数据格式。
- 手动导入导出格式。
- 应急备份可读格式。

## 3. 推荐方案：SQLite

正式环境第一阶段建议使用 SQLite。原因如下：

- 单文件数据库，部署和备份简单。
- 支持事务，能保证月结、奖金快照等多表写入一致性。
- 支持索引和约束，便于防止重复日期、重复月结、无效外键等问题。
- 运维成本低，适合单科室、单服务实例部署。
- 后续可以通过 ORM 或迁移脚本演进表结构。
- 未来如需升级 PostgreSQL，可以复用大部分数据模型和迁移思路。

建议使用：

- SQLite 数据库文件：`data/schedule.db` 或正式服务器路径 `/var/lib/my-working-schedule/schedule.db`。
- 数据访问层：保留现有 Storage Adapter 抽象，新增 SQLite 实现。
- 迁移管理：优先选择 Drizzle ORM 的迁移能力，或先使用轻量 SQL migration 文件。

## 4. 建议数据模型

正式库表建议按业务边界拆分：

| 表名 | 用途 |
| --- | --- |
| `staff` | 人员资料，包含工号、姓名、人员类型、是否管理员、启用状态、排序 |
| `shifts` | 班次配置，包含名称、简称、颜色、是否计出勤、系数、启用状态、排序 |
| `holidays` | 节假日配置，包含日期、名称、是否影响满勤 |
| `schedule_entries` | 排班记录，按人员和日期唯一 |
| `schedule_entry_shifts` | 排班记录与班次的多对多明细，支持一天两个班次 |
| `monthly_settlements` | 月结主表，包含月份、日期范围、奖金池、月结时间、总系数 |
| `monthly_settlement_rows` | 月结快照明细，保存人员、工号、班次、加班、系数、奖金金额 |
| `app_settings` | 系统参数，例如默认满勤班次数 |
| `audit_logs` | 操作审计，记录关键写操作、操作者、时间、变更对象和摘要 |
| `users` | 后续账号体系，替代单一管理员密码 |

月结和奖金数据必须保存快照，不应只在查询时实时计算。这样可以保证历史月结结果不会因为后续人员、班次系数或节假日配置调整而改变。

## 5. 数据一致性要求

SQLite 阶段建议满足以下约束：

- `staff.job_id` 唯一。
- `holidays.date` 唯一。
- `schedule_entries` 对 `staff_id + date` 唯一。
- `monthly_settlements.month` 唯一。
- 已月结月份禁止修改排班记录。
- 创建月结时，月结主表和月结明细必须在同一事务中写入。
- 删除或取消月结时，月结主表和明细必须在同一事务中删除。
- 班次禁用不应删除历史排班记录，只影响新增排班选择。
- 人员停用不应删除历史排班记录，只影响新增排班选择和显示标记。

## 6. 备份与恢复

正式环境必须配套备份策略：

- 每日自动备份 SQLite 数据库文件。
- 备份文件建议按日期命名，例如 `backups/schedule-2026-06-18.db`。
- 至少保留最近 30 天备份。
- 重要操作前支持手动备份，例如月结前、系统升级前。
- 恢复流程需要明确：停止服务、替换数据库文件、启动服务、执行健康检查。
- 后续可增加“导出 JSON”功能，便于人工检查和跨版本迁移。

如果 SQLite 使用 WAL 模式，备份时需要使用 SQLite 官方备份接口，或在备份前执行 checkpoint，避免只复制主数据库文件导致数据不完整。

### Linux SQLite 维护工具

SQLite 是嵌入式文件数据库，不需要像 MySQL 或 PostgreSQL 一样启动独立数据库服务。正式环境需要长期运行的是当前 Web/API 应用进程；SQLite 侧需要维护的是数据库文件、目录权限、迁移、备份、恢复和完整性检查。

因此后续实现阶段建议在根目录新增 `tools/`：

```text
tools/
  sqlite-service.sh
  README.md
```

`tools/sqlite-service.sh` 作为 Linux 下 SQLite 数据库文件维护入口，建议提供：

- `install`：检查 `sqlite3` CLI、创建数据目录和备份目录。
- `init`：初始化 SQLite 数据库结构。
- `migrate`：从 `storagePath` 指向的 JSON 文件迁移到 SQLite。
- `backup`：生成 SQLite 数据库备份。
- `restore <backup-file>`：从备份恢复数据库，恢复前自动备份当前数据库。
- `status`：查看数据库路径、文件大小、更新时间和备份目录。
- `check`：执行 SQLite 完整性检查和核心业务表检查。

正式 Linux 路径建议：

- 数据库文件：`/var/lib/my-working-schedule/schedule.db`。
- 备份目录：`/var/backups/my-working-schedule`。

这些路径应允许通过 `SCHEDULE_SQLITE_PATH` 和 `SCHEDULE_BACKUP_PATH` 覆盖，并确保服务运行用户拥有读写权限。

## 7. 迁移路线

建议按以下阶段推进：

### 阶段一：保留 JSON，新增 SQLite 存储实现

- 保留当前 API 和前端调用方式。
- 新增 SQLite Storage Adapter。
- 增加 JSON 到 SQLite 的一次性导入脚本，读取 `storagePath` 指向的业务数据文件，而不是导入整个 `server.local.json`。
- 增加 SQLite 到 JSON 的导出脚本。
- 增加 `tools/sqlite-service.sh`，为 Linux 正式环境提供安装检查、初始化、迁移、备份、恢复、状态和完整性检查入口。
- 测试覆盖当前所有读写接口。

### 迁移保障：历史 JSON 用户数据

切换到 SQLite 时，必须保证当前 JSON 本地文件中已经新增的真实用户数据可以完整迁移，不能只导入仓库种子数据。

当前需要迁移的源数据是 `storagePath` 指向的业务 JSON 文件，通常是：

```text
data/app-data.local.json
```

不要把 `data/app-data.json` 当作正式迁移源，除非明确要从种子数据重新初始化。`data/app-data.json` 只代表仓库默认数据，不包含用户在页面上新增或修改后的真实数据。

#### 迁移范围

JSON 到 SQLite 迁移脚本必须覆盖以下数据：

- `staff`：人员资料，包括工号、姓名、人员类型、管理员标记、启用状态和排序。
- `shifts`：班次配置，包括名称、简称、颜色、是否计出勤、系数、启用状态和排序。
- `holidays`：节假日配置，包括日期、名称和是否影响满勤。
- `scheduleEntries`：排班记录，包括人员、日期、班次数组和备注。
- `monthlySettlements`：月结与奖金快照，包括月结主信息和每个人员的快照明细。
- `settings`：系统基础设置。

迁移脚本需要兼容当前存储层已经支持的历史字段补齐逻辑，例如旧月结明细中缺少 `staffJobId` 或 `overtimeShifts` 时，应按现有规范补齐默认值后再导入。

#### 安全迁移流程

正式切换 SQLite 前建议按以下步骤执行：

1. 停止服务，避免迁移过程中继续写入 JSON。
2. 确认 `server.local.json` 或 `SCHEDULE_DATA_PATH` 指向的业务 JSON 文件路径。
3. 备份源 JSON 文件，例如 `backups/app-data-before-sqlite-YYYYMMDD-HHmmss.json`。
4. 校验源 JSON 结构，确认能被当前服务端 `AppData` 规则识别。
5. 创建 SQLite 数据库和表结构。
6. 在一个 SQLite 事务中导入全部业务数据。
7. 导入完成后执行数量校验和抽样校验。
8. 将服务配置切换为 SQLite 存储。
9. 启动服务并执行健康检查、页面读取检查和关键业务检查。
10. 保留迁移前 JSON 备份一段时间，确认稳定后再归档。

如果任一步失败，应停止切换，保留原 JSON 存储配置，并用迁移前备份恢复。

#### 数据不丢失校验

迁移脚本完成后必须输出校验报告，至少包含：

| 数据类型 | 校验方式 |
| --- | --- |
| 人员 | JSON `staff.length` 与 SQLite `staff` 行数一致 |
| 班次 | JSON `shifts.length` 与 SQLite `shifts` 行数一致 |
| 节假日 | JSON `holidays.length` 与 SQLite `holidays` 行数一致 |
| 排班记录 | JSON `scheduleEntries.length` 与 SQLite `schedule_entries` 行数一致 |
| 单日班次明细 | JSON 所有 `shiftIds` 总数与 SQLite `schedule_entry_shifts` 行数一致 |
| 月结主表 | JSON `monthlySettlements.length` 与 SQLite `monthly_settlements` 行数一致 |
| 月结明细 | JSON 所有月结 `rows` 总数与 SQLite `monthly_settlement_rows` 行数一致 |
| 系统设置 | 默认满勤班次数和版本号一致 |

除数量校验外，还需要做抽样校验：

- 随机人员的工号、姓名、类型和启用状态一致。
- 随机班次的简称、颜色、系数和启用状态一致。
- 随机日期的人员排班和班次数组一致。
- 已月结月份的奖金池、总系数、人员奖金金额和排除原因一致。

校验不通过时，迁移结果不能启用。

#### 回滚要求

SQLite 切换必须支持回滚：

- 保留迁移前 JSON 备份。
- 保留原 JSON 存储配置。
- 切换失败时，可以重新把服务配置指回 JSON 文件。
- 如果 SQLite 已经启动但发现数据异常，应停止服务，恢复 JSON 配置，再排查迁移脚本。

在确认 SQLite 稳定运行前，不应删除原始 `data/app-data.local.json` 或迁移前备份。

### 阶段二：正式环境默认使用 SQLite

- 通过 `SCHEDULE_STORAGE_DRIVER=sqlite` 或配置文件选择存储驱动。
- 正式部署默认写入 SQLite。
- 开发环境仍可使用 JSON，降低本地调试门槛。
- 增加启动时数据文件路径检查和备份目录检查。

### 阶段三：补齐正式化能力

- 增加自动备份和手动备份命令。
- 增加操作审计日志。
- 增加账号体系和角色权限。
- 增加数据导入导出页面或管理命令。
- 增加月结前快照和恢复演练文档。

### 阶段四：按规模评估 PostgreSQL

当出现以下情况时，再考虑升级 PostgreSQL：

- 多科室或多院区集中部署。
- 多服务实例同时写入。
- 需要复杂权限、报表和集中运维。
- 需要接入医院统一账号、统一审计或数据平台。

## 8. 不推荐 etcd 作为主业务存储

早期技术方案中提到过 Go + Gin + etcd 作为正式化方向。经当前业务评估，etcd 更适合作为分布式配置、服务发现和协调组件，不适合作为排班、月结、奖金、审计这类关系型业务数据的主存储。

因此建议调整为：

- 单机正式部署：SQLite。
- 多实例或集中部署：PostgreSQL。
- etcd：仅在未来需要服务协调或分布式配置时再评估。

## 9. 验收标准

存储优化完成后，应满足：

- 重启服务后，人员、班次、节假日、排班、月结和奖金数据不丢失。
- 创建月结失败时，不产生半条月结数据。
- 已月结月份不能修改排班。
- 数据库文件可备份、可恢复。
- Linux 下可通过 `tools/sqlite-service.sh` 维护 SQLite 数据库文件。
- 从当前 JSON 数据可以平滑迁移到 SQLite。
- 迁移脚本能读取 `storagePath` 指向的真实业务 JSON，而不是只读取仓库种子数据。
- 迁移后人员、班次、节假日、排班、月结和月结明细数量校验一致。
- 迁移后抽样数据校验一致，并输出迁移校验报告。
- 迁移失败时可以回滚到原 JSON 存储。
- 所有现有 API 测试继续通过。
- 新增迁移、备份、恢复测试覆盖关键路径。
