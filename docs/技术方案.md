# Web 版本技术方案

## 文档状态

本文与当前执行计划对齐，描述一期落地方案和后续正式架构演进方向。一期优先实现排班管理闭环，采用轻量 Web 应用；当前开发和正式单机部署均使用 SQLite 文件数据库。Go + Gin、PostgreSQL 等仅作为更大规模场景下的后续评估方向，不作为当前正式化第一步。

## 1. 一期技术栈

一期采用：

*   **前端**：Vue 3 + Vite + TypeScript + Element Plus。
*   **后端**：Express API。
*   **存储**：SQLite 文件数据库。JSON 不再作为运行时存储，也不再提供旧 JSON 导入或 SQLite 导出 JSON 工具。
*   **测试**：Vitest、Supertest、Playwright。

该组合用于快速完成护士长真实排班流程，包含人员、班次、节假日、排班记录、周统计、全屏和打印。前端只通过 API 访问数据，不直接读写 SQLite 文件；本地运行数据库不进入 Git，便于验证后重置。

## 2. 一期模块划分

*   **排班工作台**：提供当前自然周排班表、日期选择、周范围定位、班次画笔、单元格编辑弹窗、全屏和打印入口。
*   **周统计模块**：按自然周（周一至周日）统计每人出勤班次数、加班班次数和总系数，并显示满勤标准、节假日扣减和护士长绩效单独核算说明。
*   **人员管理**：维护护士、文员、护士长、工号、管理员权限、启用状态和排序。
*   **班次管理**：维护班次名称、简称、颜色、是否计出勤、系数、启用状态和排序。
*   **节假日管理**：内置默认节假日，并允许管理员维护日期、名称和是否影响满勤。
*   **打印视图**：提供月排班表和周统计表的 CSS Print 输出。
*   **账号权限与审计**：提供登录页、基础角色权限、会话、退出登录和关键写操作审计。

## 3. 一期 API 与数据边界

一期 API 采用简化路径：

*   `GET /api/health`：健康检查。
*   `GET /api/data`：读取排班系统数据。
*   `POST /api/auth/login`：账号密码登录。
*   `GET /api/auth/me`：获取当前登录用户。
*   `POST /api/auth/logout`：退出登录。
*   `PUT /api/auth/password`：登录用户修改自己的密码。
*   `GET /api/users`：读取账号列表，管理员可访问。
*   `PUT /api/users/:id`：新增或更新账号，管理员可访问。
*   `GET /api/audit-logs`：读取最近审计日志，支持账号、操作类型和关键词筛选，管理员可访问。
*   `POST /api/admin/session`：兼容旧管理密码入口，内部映射到默认管理员会话。
*   `PUT /api/data/staff/:id`：新增或更新人员。
*   `PUT /api/data/shift/:id`：新增或更新班次。
*   `PUT /api/data/holiday/:id`：新增或更新节假日。
*   `PUT /api/data/schedule-entry`：保存某人某日的排班记录。

默认管理员账号为 `admin`。管理员初始密码由服务端配置提供，优先读取 `SCHEDULE_ADMIN_PASSWORD` 环境变量，其次读取 `config/server.local.json` 或 `SCHEDULE_CONFIG_PATH` 指定的配置文件。SQLite 模式下会将默认管理员密码哈希保存到 `users` 表，并在服务启动时按当前配置刷新默认管理员密码。

数据分为：

*   `staff`：人员资料。
*   `shifts`：班次配置。
*   `holidays`：节假日配置。
*   `scheduleEntries`：排班记录。
*   `settings`：默认满勤班次数和版本号。

## 4. 一期核算规则

一期核算按“班次”而不是自然日计算。单人单日最多两个班次。班次是否计出勤、系数是多少，由班次配置决定。

默认每周满勤标准为 5 个计出勤班次。若当周存在配置为影响满勤的节假日，则按节假日数量扣减满勤标准。加班班次数按以下公式计算：

```text
加班班次数 = max(0, 计出勤班次数 - 当周满勤标准)
```

护士和文员按相同规则统计出勤、加班和总系数。护士长参与出勤和加班统计，但绩效系数标注为单独核算，不并入普通人员绩效系数。

## 5. 后续正式架构

当一期流程验证稳定后，可演进到正式架构。正式环境存储优先参考 [正式环境存储优化方案](./正式环境存储优化方案.md)，第一阶段采用 SQLite；正式部署步骤参考 [正式部署运行手册](./正式部署运行手册.md)。Go/Gin 或 PostgreSQL 仅作为更大规模、集中部署或多实例场景下的后续评估方向。

*   **Express + systemd**：当前正式单机部署继续使用 Express API，由 systemd 管理后台进程。
*   **SQLite**：当前运行时存储，提供事务、约束、备份和稳定的单机持久化能力。
*   **PostgreSQL**：在多科室、多实例或集中部署时作为后续升级方向。
*   **Go + Gin**：在需要重构正式 API 或规则引擎时再评估。
*   **规则引擎**：抽象节假日、满勤、系数、奖金等可调整规则。
*   **账号、权限与审计**：替代单一管理密码，记录关键操作。
*   **部署运维**：补齐 systemd、Nginx、备份、恢复、日志和上线检查流程。

后续接口可以向 `/api/v1/staff`、`/api/v1/schedule`、`/api/v1/calculation` 等正式 RESTful 路径演进，但一期执行计划先保持轻量。
