# 账号绑定人员后的细粒度权限设计

## 背景

系统已经完成账号登录、角色权限、审计日志，以及账号与人员档案的一对一绑定。当前绑定关系主要用于标识账号对应的人员，但尚未真正影响排班和月结操作权限。

本设计的目标是让“账号绑定人员”和“可管理人员范围”成为正式权限边界：所有登录用户可以查看全科排班，但写入排班、月结和系统配置时必须受到后端权限校验约束。

## 已确认规则

- 采用集中权限服务方案，避免权限判断散落在接口和页面代码中。
- 普通护士、文员等查看类账号允许查看全科排班，但不能编辑。
- 护士长、排班员账号允许查看全科排班，但只能编辑自己可管理人员范围内的排班。
- 月结与奖金允许查看全科汇总，但护士长、排班员只能创建或删除自己可管理人员范围内的月结记录。
- 新增账号默认不管理任何人，需要管理员显式配置可管理人员名单。
- 排班页面中无权限编辑的单元格保持正常显示，点击时静默只读，不弹错误提示。
- 后端是最终安全边界。任何越权写入排班、月结或配置的请求必须返回 `403`。

## 权限模型

账号权限由三部分组成：

1. 角色
   - `admin`：全系统管理权限。
   - `scheduler`：排班和月结操作权限，但受可管理人员名单限制。
   - `viewer`：全科只读权限。

2. 绑定人员
   - 每个账号可以绑定一个人员档案。
   - 绑定人员用于标识账号本人，并让审计日志可以关联到具体人员。
   - 绑定人员不自动授予编辑权限。

3. 可管理人员名单
   - 每个账号可以配置多个可管理人员。
   - 新账号默认名单为空。
   - `scheduler` 只能编辑名单内人员的排班，并只能创建或删除名单内人员的月结记录。
   - `viewer` 即使配置了可管理人员名单，也保持只读。
   - `admin` 默认拥有全部人员管理权限，不依赖可管理人员名单。

## 数据结构

保留现有 `users.staff_id` 字段，用于表示账号绑定的本人档案。

新增 SQLite 关联表 `user_managed_staff`：

| 字段 | 含义 |
| --- | --- |
| `user_id` | 账号 ID |
| `staff_id` | 可管理人员 ID |
| `created_at` | 配置时间 |
| `created_by` | 配置该关系的管理员账号 ID |

约束：

- `user_id` 外键关联 `users.id`。
- `staff_id` 外键关联 `staff.id`。
- `(user_id, staff_id)` 建唯一索引，避免重复配置。

迁移策略：

- 老库升级时只新增空表，不自动填充可管理人员。
- 已有 `admin` 账号不受影响，仍默认管理全部人员。
- 已有 `scheduler` 账号升级后默认不能编辑任何人员，需要管理员显式配置。
- 已停用人员如果已经在可管理名单中，编辑账号时保留显示并标记“已停用”；新增选择时只允许选择启用人员。

## 后端行为

新增统一权限判断模块，提供清晰的权限语义：

- 是否可以读取全科数据。
- 是否可以编辑指定人员的排班。
- 是否可以创建或删除指定人员的月结。
- 是否可以维护系统配置、账号和审计日志。

API 权限边界：

- `GET /api/data` 从公开接口收敛为登录后访问。
- 排班写入接口校验当前用户是否能编辑目标人员。
- 月结创建和删除接口校验当前用户是否能操作目标人员范围。
- 账号、人员、班次、节假日、审计日志接口继续保持管理员权限。
- 越权写入统一返回 `403`，并记录审计日志。

## 前端行为

所有登录用户都可以查看全科排班和允许公开的统计信息。

排班页面：

- `admin` 可以编辑全部人员排班。
- `scheduler` 只能编辑可管理人员名单内人员的排班。
- `viewer` 只能查看。
- 无权限单元格保持正常显示，但点击无变化。
- 首页说明栏补充权限说明，说明“可查看全科排班，编辑范围由账号权限决定”。

月结与奖金页面：

- 所有登录用户可以查看全科汇总。
- `admin` 可以操作全部人员月结。
- `scheduler` 只能创建或删除可管理人员名单内人员的月结。
- `viewer` 只读。

账号维护页面：

- 账号列表新增“可管理人员”列。
- 新建和编辑账号表单新增“可管理人员”多选框。
- `admin` 角色提示“管理员默认管理全部人员”，不需要配置名单。
- `viewer` 的可管理人员配置隐藏或置灰，避免误解。
- 人员选项展示姓名和工号。
- 多选默认不选任何人。
- 已停用但保留在名单中的人员继续显示，并标记“已停用”。

## 审计日志

以下行为需要记录审计：

- 新建账号时的绑定人员和可管理人员摘要。
- 修改账号时的绑定人员变化和可管理人员名单变化。
- 越权写入排班或月结被拒绝。
- 合法排班写入和月结操作继续记录现有业务审计。

审计摘要中应包含人员姓名和工号，方便管理员追溯。

## 测试设计

权限服务单元测试：

- `admin` 可管理全部人员。
- `scheduler` 只能管理名单内人员。
- `viewer` 永远只读。
- 未绑定人员账号仍可登录，但不影响可管理范围。
- 停用人员已有管理关系可保留，但不能新增配置。

API 权限测试：

- 未登录访问 `/api/data` 被拒绝。
- `viewer` 写排班返回 `403`。
- `scheduler` 写名单内人员排班成功。
- `scheduler` 写名单外人员排班返回 `403`。
- `scheduler` 创建或删除名单内月结成功。
- `scheduler` 创建或删除名单外月结返回 `403`。
- 账号维护接口校验重复、停用和非法人员。
- 越权失败写入审计日志。

页面验收：

- 登录后可以进入系统并查看全科排班。
- 管理员可以在账号维护页面配置可管理人员。
- 排班员只能修改可管理人员范围内的排班。
- 普通查看账号能看全科排班，但不能编辑。
- 首页说明栏展示权限说明。
- 审计日志能看到账号管理范围变更、越权拦截和排班写入记录。

## 非目标

本阶段不实现科室或小组模型。

本阶段不做历史审计回填。审计日志只从账号绑定和权限配置生效后开始记录可关联人员信息。

本阶段不限制普通账号查看全科排班，因为已确认查看权限允许全科可见。
