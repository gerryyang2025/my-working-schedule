# P0-B 账号与人员档案绑定设计

## 背景

当前系统已经具备登录页、账号维护、基础角色权限和审计日志能力。账号数据存储在 SQLite 的 `users` 表，人员档案存储在 `staff` 表，两者尚未建立关联。P0-B 的目标是补上账号与人员档案之间的一对一身份绑定，为后续“护士长辖区权限”“个人只看自己排班”“更细粒度角色”提供基础骨架。

本阶段只设计和实现绑定关系，不改变现有 `admin`、`scheduler`、`viewer` 的访问范围。

## 已确认决策

- 一个账号最多绑定一个人员档案。
- 一个人员最多被一个账号绑定。
- 系统账号可以不绑定人员，例如默认 `admin`。
- 本阶段绑定后不改变现有权限范围。
- 绑定入口放在“系统配置 > 账号”。
- 账号显示名和人员姓名独立维护，绑定后不自动同步。
- 不回填历史审计日志。
- 新增绑定时只允许选择启用人员。
- 如果人员后续被停用，账号保留绑定关系，并在页面标记“人员已停用”。
- 现有账号迁移时默认不自动绑定，由管理员后续手动维护。
- 本阶段不强制校验人员类型和账号角色，只显示提醒。
- 正式存储已切换为 SQLite，本阶段不考虑 JSON 业务存储兼容；后续可单独删除废弃 JSON 存储方案。

## 方案选择

采用方案 A：在 `users` 表直接新增 `staff_id` 字段。

```text
users.staff_id text nullable unique references staff(id)
```

选择理由：

- 当前目标是一对一可选绑定，直接字段最清晰。
- 登录态和账号列表读取时可以自然带出绑定人员 ID。
- `unique(staff_id)` 能直接保证一个人员最多绑定一个账号。
- 后续护士长辖区或多人范围权限应单独建模，例如新增 `user_scopes` 或 `head_nurse_scopes`，不混入基础绑定。

不采用绑定表的原因：

- 本阶段不需要绑定历史、多人员绑定或代理关系。
- 单独绑定表会增加接口和查询复杂度。
- 未来如果需要辖区或多人授权，可以在基础一对一绑定之上新增权限范围表。

## 数据结构

SQLite schema version 从当前版本 `2` 递增到 `3`。

对 `users` 表新增字段：

```sql
alter table users add column staff_id text;
```

既有 SQLite 表通过 `alter table` 增加字段。由于 SQLite 对既有表追加外键约束的能力有限，迁移后的旧库主要依赖应用层校验和唯一索引保证绑定正确性；新建数据库的 `users` 表可以直接声明 `staff_id text references staff(id)`。

新增唯一索引：

```sql
create unique index if not exists idx_users_staff_id_unique
on users(staff_id)
where staff_id is not null;
```

说明：

- `staff_id = null` 表示未绑定人员。
- SQLite 允许唯一索引中存在多条 `null`，因此多个未绑定账号不会冲突。
- 新建数据库应直接创建包含 `staff_id text references staff(id)` 的 `users` 表和唯一索引。
- 既有 SQLite 数据库迁移后，已有账号保持 `staff_id = null`。

## 后端模型与接口

扩展账号相关类型：

- `AuthUser`
- `PublicAuthUser`
- `ManagedAuthUser`
- `SaveAuthUserInput`

新增字段：

```ts
staffId: string | null
```

接口行为：

- `POST /api/auth/login` 返回 `user.staffId`。
- `GET /api/auth/me` 返回 `user.staffId`。
- `GET /api/users` 返回每个账号的 `staffId`。
- `PUT /api/users/:id` 接收 `staffId`，空值表示解绑。

保存账号时的校验：

- 只有 `admin` 可以维护账号和绑定关系，沿用现有账号维护权限。
- `staffId = null` 时允许保存。
- `staffId != null` 时，人员必须存在。
- 新绑定人员必须是启用状态。
- 如果账号原本已绑定某个人员，且该人员后来被停用，保存账号时允许继续保留这个原绑定关系。
- 如果账号换绑到另一个人员，目标人员必须是启用状态。
- 目标人员不能已经绑定给其他账号。
- 人员类型和账号角色不做强制约束。
- 默认 `admin` 允许不绑定人员。

## 页面入口与交互

入口保持在“系统配置 > 账号”，不新增页面。

账号列表增加“绑定人员”列：

- 已绑定启用人员：`姓名 / 工号`
- 未绑定：`未绑定`
- 已绑定但人员停用：`姓名 / 工号（已停用）`

手机端账号卡片同步显示绑定人员信息。

账号表单增加“绑定人员”下拉框：

- 支持清空，清空后保存为未绑定。
- 下拉只列出启用人员，以及当前账号已经绑定的人员。
- 选项显示 `姓名 / 工号 / 人员类型`。
- 若人员类型和账号角色看起来不匹配，不阻止保存，只提示：本阶段仅建立身份绑定，不会改变账号权限。

保存场景：

- 新建账号时可以绑定人员，也可以暂不绑定。
- 编辑账号时可以绑定、换绑、解绑。
- 如果后端返回人员已被其他账号绑定，页面显示错误提示。
- 如果人员后续被停用，账号页面保留绑定并标记“已停用”，管理员可手动解绑或换绑。

## 权限规则

本阶段不改变现有角色权限：

- `admin` 仍可维护人员、班次、节假日、账号、排班、月结和审计。
- `scheduler` 仍可维护排班和月结。
- `viewer` 仍只读。

`staffId` 只作为身份基础字段，不影响当前数据读取、排班保存、月结保存。后续“个人只看自己排班”“护士长辖区权限”再基于 `currentUser.staffId` 扩展。

## 审计规则

不回填历史审计日志。

账号保存仍记录 `user.save`。摘要建议补充绑定状态：

- `保存账号：viewer1，未绑定人员`
- `保存账号：viewer1，绑定人员：李护士(100001)`
- `保存账号：viewer1，绑定人员：李护士(100001，已停用)`

审计日志表本阶段不新增 `staff_id` 字段，避免把审计历史和当前绑定关系混在一起。后续如果需要强追溯，可单独设计审计快照字段。

## 迁移策略

正式环境以 SQLite 为准。

迁移步骤：

1. 检查 `users` 表是否已有 `staff_id` 字段。
2. 若不存在，执行 `alter table users add column staff_id text`。
3. 创建 `idx_users_staff_id_unique` 部分唯一索引。
4. 将 schema version 记录为 `3`。
5. 不自动匹配账号和人员。
6. 不回填历史审计日志。

迁移后验证：

- 既有账号均可正常登录。
- `/api/users` 中既有账号的 `staffId` 为 `null`。
- 管理员可以在账号页面手动绑定人员。

## 错误处理

后端错误建议：

- 人员不存在：`绑定人员不存在`
- 人员已停用：`只能绑定启用人员`
- 人员已被其他账号绑定：`该人员已绑定其他账号`
- 账号名重复、最后一个管理员保护等现有错误保持不变。

前端只展示后端返回的错误信息，不在前端重复实现关键约束。前端的下拉筛选和提示只用于提升操作体验。

## 测试计划

后端测试：

- SQLite schema 迁移后 `users.staff_id` 存在。
- `staff_id` 唯一约束生效。
- 既有账号迁移后 `staffId = null`。
- 保存账号可以绑定启用人员。
- 保存账号可以解绑人员。
- 保存账号不能绑定不存在人员。
- 保存账号不能新绑停用人员。
- 保存账号不能绑定已被其他账号绑定的人员。
- 已绑定人员后续停用，账号仍可读取并返回 `staffId`。
- 登录、`/api/auth/me`、`/api/users` 均返回 `staffId`。
- 本阶段不改变 `viewer`、`scheduler` 的现有权限测试结果。

前端测试：

- 账号列表显示绑定人员姓名和工号。
- 未绑定账号显示 `未绑定`。
- 已绑定停用人员显示停用标记。
- 账号表单可以选择启用人员。
- 账号表单可以清空绑定。
- 手机端账号卡片显示绑定人员信息。
- 保存账号时提交 `staffId`。

文档测试：

- 更新功能跟进清单，标记 P0-B 的设计和实现状态。
- 更新正式部署运行手册或使用说明，补充管理员如何绑定账号和人员档案。

## 非目标

本阶段不实现：

- 个人只看自己排班。
- 护士长辖区权限。
- 科室、班组、楼层等组织结构。
- 多人员绑定或代理账号。
- 绑定历史记录。
- 历史审计回填。
- 删除 JSON 存储方案。该清理可以作为后续单独任务。

## 后续扩展

完成 P0-B 后，下一步可以基于 `users.staff_id` 设计：

- `viewer` 只查看本人排班。
- 护士长绑定本人档案后，再配置辖区范围。
- 审计日志按绑定人员筛选。
- 排班页面根据当前账号身份默认定位到本人或辖区人员。
