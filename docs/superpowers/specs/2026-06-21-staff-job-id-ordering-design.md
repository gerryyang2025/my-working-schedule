# Staff Job ID Ordering Design

## 背景

当前人员展示顺序主要依赖人员档案中的 `sortOrder` 字段。每次新增人员后都需要手动维护排序 ID，否则排班表、统计表和打印内容的顺序不符合实际查看习惯。

用户确认采用方案 A：界面上不再让用户维护人员排序 ID，人员展示统一默认按工号排序，并在排班表第一列增加当前行号。

## 目标

- 排班信息中的人员排列顺序默认按工号升序显示。
- 新增人员时不再要求用户额外指定排序 ID。
- 排班表第一列显示当前可见人员的行号，行号只用于查看，不写入人员数据。
- 周统计、月结与奖金、打印视图等人员列表尽量与排班表保持一致顺序。

## 非目标

- 本次不删除数据库或领域模型中的 `sortOrder` 字段。
- 本次不修改班次排序，班次仍使用现有 `sortOrder`。
- 本次不做拖拽排序、手动置顶、按人员类型分组等复杂排序能力。

## 排序规则

新增共享的人员排序规则，所有人员展示场景优先复用：

1. 按 `jobId` 使用自然排序升序排列，支持数字工号和带前缀工号。
2. 工号相同时按姓名排序。
3. 姓名仍相同时按人员 `id` 排序，保证稳定输出。

示例顺序：

```text
000228
000815
100001
200001
```

## 展示范围

以下页面和输出应改为按工号排序：

- 排班表人员行。
- 周统计人员行。
- 月结与奖金人员行。
- 打印周表和打印月表中的人员行。
- 人员配置列表。
- 账号绑定人员、可管理人员下拉选项。

停用人员仍遵循现有显示规则：如果停用人员在当前展示范围内有历史排班或月结数据，仍保留显示，并参与同一套工号排序。

## 排班表行号

排班表新增最左侧行号列：

- 表头显示“序号”。
- 每一行显示当前可见人员列表的 1-based 行号。
- 行号不保存到数据库，不参与人员档案、排班记录、统计或打印计算。
- 行号只反映当前表格显示顺序；如果未来筛选人员，行号跟随筛选后的可见顺序重新计算。

移动端横向滚动时，行号列和人员列都需要固定在左侧。行号列保持窄列，人员列继续根据姓名宽度自适应。

## 人员配置页面

人员配置页不再显示人员 `sortOrder`：

- 移除人员表格中的“排序”列。
- 移除人员表单中的排序输入框。
- 新增人员时内部仍给出默认 `sortOrder`，仅用于兼容当前 API 和旧数据结构。
- 编辑旧人员时不再允许通过 UI 修改 `sortOrder`。

文档中“人员维护”描述需要同步，不再把排序作为用户需要维护的配置项。

## 数据与兼容

本次保留 `StaffMember.sortOrder`、服务端校验、SQLite 字段和历史数据：

- 避免引入数据库迁移风险。
- 避免影响已有 API 请求结构和测试夹具。
- 后续如果确认完全不需要该字段，可单独做一次字段废弃和 schema 清理。

前端保存人员时仍随请求提交一个内部默认 `sortOrder` 值，服务端无需新增兼容分支。

## 测试策略

- 为共享排序工具增加单元测试：数字工号、带前缀工号、相同工号按姓名和 id 兜底。
- 更新排班表测试：验证人员按工号排序，行号列显示 1、2、3。
- 更新人员配置测试：验证人员排序输入不再出现，新增人员不需要手动填写排序。
- 更新周统计、月结与奖金、打印视图相关测试，确认人员顺序按工号排列。
- 保留现有 lint 和全量测试作为回归验证。

## 验收标准

- 新增人员时，界面不再出现人员排序 ID 输入项。
- 排班表人员按工号顺序显示。
- 排班表最左侧有序号列，序号从 1 开始连续显示。
- 移动端横向滚动时，序号列和人员列都固定在左侧。
- 周统计、月结与奖金、打印周表和打印月表中的人员顺序与排班表一致。
- 不影响班次排序、排班保存、月结、账号绑定和权限校验。
